# AxureExport 技术规则

本文定义这套 Axure 兼容原型框架的通用技术基线。

目标是把稳定、不随项目视觉风格变化的部分沉淀下来，供后续同类项目复用。

## 1. 项目定位

这套工程的目标不是普通前端站点，而是一套：

- 可本地直接打开
- 可快速替换产品信息和页面内容
- 可复用通用后台壳
- 可重新打包并上传摹客

的演示原型框架。

## 2. 当前真实架构

当前项目采用的是“真实 Axure 多页面壳 + 共享业务配置 + 通用渲染增强”方案。

### 2.1 入口结构

- `index.html`
  兼容保留页，可用于本地调试
- `仪表盘.html`
  仪表盘页面入口
- `视频流分析.html`
  视频流分析页面入口
- `视频流新建任务.html`
  视频流新建任务页面入口
- `start.html`
  Axure 播放器入口

页面访问方式统一为真实页面地址，例如：

```text
仪表盘.html
```

摹客左侧页面树读取的是 `data/document.js` 里的 sitemap，而不是浏览器 hash 路由。

### 2.2 页面结构说明

页面结构以真实 Axure 导出结构为准。

具体原则如下：

- 每个业务模块必须对应一个真实页面文件，而不是 `index.html#xxx`
- 摹客左侧树只认 `data/document.js` 中的 sitemap 和真实页面 url
- 页面 html、`files/<页面>/data.js`、`files/<页面>/styles.css` 必须保持 Axure 导出兼容结构
- 我们自己的业务壳、导航、卡片、表格能力，通过共享脚本和共享样式追加进去
- 页面基座优先保持接近 Axure 原始导出，只做最小增强，不要整体改写

当前页面树结构如下：

- 首页
  - 仪表盘
  - 调度中心
- 接入管理
  - 视图源接入
  - 国标设备
- 任务管理
  - 视频流分析
  - 视频流新建任务

### 2.3 主要维护文件

- `resources/scripts/prototype.config.js`
  基础配置层。产品信息、导航、页面注册表、mock 存储开关优先都在这里改
- `resources/scripts/prototype.page-loader.js`
  页面模块加载层。负责按 `pageRegistry` 加载独立页面定义文件
- `resources/scripts/pages/*.page.js`
  页面定义层。每个页面独立维护自己的页面结构、文案、示例数据与页面特有交互
- `resources/scripts/prototype.style-loader.js`
  页面样式加载层。负责按页面定义动态挂载 `resources/css/pages/*.css`
- `resources/scripts/mock-store.js`
  mock 数据访问层。负责浏览器存储读写、页面 mock 数据重置与导入导出
- `resources/scripts/prototype.utils.js`
  通用工具层。负责模板替换、转义、查找归一化、图表路径计算等纯函数
- `resources/scripts/prototype.shell.js`
  共享壳层。负责壳布局、导航、topbar、toast 与共享渲染片段
- `resources/scripts/prototype.dashboard.js`
  dashboard 渲染层。负责 dashboard 页面和各类 dashboard panel 的渲染
- `resources/scripts/prototype.table.js`
  table 渲染层。负责 table 页面渲染和表格筛选绑定
- `resources/scripts/prototype.components.js`
  共享组件装配层。负责把 `shell / dashboard / table` 组装成统一组件接口
- `resources/scripts/prototype.js`
  启动装配层。负责路由、页面加载、存储层与组件层之间的连接
- `resources/css/prototype.css`
  共享样式入口层。负责聚合 `prototype.tokens/base/shell/dashboard/table/responsive.css`
- `resources/css/pages/*.css`
  页面样式层。每个页面独立维护自己的局部样式补充

### 2.4 架构结论

当前项目是“每个模块对应一个真实 Axure 页面”的多页方案，但这些页面共享同一套框架约定：

- `prototype.config.js`
- `prototype.page-loader.js`
- `prototype.style-loader.js`
- `pages/*.page.js`
- `prototype.shell.js`
- `prototype.dashboard.js`
- `prototype.table.js`
- `prototype.js`
- `prototype.css`
- `css/pages/*.css`

后续新增页面时：

- 默认要新增真实页面 html 文件
- 默认要新增对应的 `files/<page>/data.js` 与 `files/<page>/styles.css`
- 默认在 `prototype.config.js` 里注册页面
- 默认在 `resources/scripts/pages/*.page.js` 中维护页面内容
- 默认在 `resources/css/pages/*.css` 中维护页面局部样式
- 默认通过 `data/document.js` 把页面挂到 sitemap 中
- 默认优先复制一份已验证可上传的页面壳，不要手写新的 Axure 页面结构

## 3. 技术约束

以下规则默认适用于后续所有同类项目：

- 仅使用 HTML + 原生 CSS + 原生 JS
- 不引入 Vue、React 等前端框架
- 不引入构建工具作为运行前提
- 不引入后端依赖
- 不引入网络请求作为页面渲染前提
- 页面应保持可本地直接打开和演示
- 页面数据优先使用 `resources/scripts/pages/*.page.js` 中的静态 mock seed
- 如需本地持久化 mock，可通过 `mock-store.js` 接入浏览器存储

说明：

- 当前框架已支持通过 `mock-store.js` 使用 `localStorage` / `sessionStorage`
- 默认驱动为 `localStorage`
- 若只希望当前浏览器会话内有效，可把 `prototype.config.js` 里的 `mockStore.driver` 改为 `sessionStorage`

## 4. 禁改边界

以下文件属于兼容层，尽量保持不动：

- `start.html`
- `data/document.js`
- `resources/scripts/axure/*`
- `resources/scripts/player/*`
- `plugins/*`

以下文件属于页面壳文件，通常通过复制现有页面生成，不建议随意改动结构：

- `仪表盘.html`
- `视频流分析.html`
- `视频流新建任务.html`
- `files/*/data.js`
- `files/*/styles.css`

原因：

- 这些文件用于 Axure / 摹客兼容
- 如果这些文件被大改，很容易再次出现“演示包文件已损坏”
- 页面 html 和 `files/*/data.js` 的编码、结构、脚本加载顺序都会影响摹客校验

补充规则：

- 页面 html、`data/document.js`、`files/*/data.js` 建议保持 UTF-8 BOM
- 页面壳优先复制自已验证通过的 Axure 样例页面，再追加业务增强脚本
- 不要把整个页面 html 改成纯自定义壳，否则很容易失去 Axure 导出兼容性

## 7. 数据与配置规则

当前项目的数据组织规则如下：

- 产品信息、导航结构、页面注册表统一放在 `prototype.config.js`
- 每个页面的结构、文案、示例数据独立放在 `resources/scripts/pages/*.page.js`
- 页面模块可选暴露 `setup(context)`，用于挂载页面特有交互
- 每个页面的局部样式独立放在 `resources/css/pages/*.css`
- 页面模块中的数据同时作为 mock seed
- 持久化后的 mock 数据优先从 `mock-store.js` 读取
- 页面内容优先通过配置表达，而不是在 `prototype.js` 中写死业务分支
- `prototype.utils.js` 只放纯函数工具
- `prototype.shell.js` 只放共享壳、导航、toast 与共享渲染片段
- `prototype.dashboard.js` 只放 dashboard 渲染
- `prototype.table.js` 只放 table 渲染与表格交互绑定
- `prototype.components.js` 只放共享组件装配，不再堆具体区块实现
- `prototype.js` 只负责启动、路由和装配逻辑
- `prototype.style-loader.js` 只负责页面样式文件挂载与切换
- `prototype.css` 只作为共享样式入口，不再直接承载大段样式实现
- `resources/css/pages/*.css` 只负责页面局部样式表达
- `data/document.js` 只负责摹客 / Axure 的页面树定义，不承担业务内容配置
- `files/<页面>/data.js` 与 `files/<页面>/styles.css` 默认视为兼容层，不作为业务主要维护入口

修改优先级：

1. 先改 `prototype.config.js`
2. 再改对应的 `resources/scripts/pages/*.page.js`
3. 再改对应的 `resources/css/pages/*.css`
4. 如需持久化能力，再看 `mock-store.js`
5. 如需新增共享壳或导航能力，再改 `prototype.shell.js`
6. 如需新增 dashboard 通用区块，再改 `prototype.dashboard.js`
7. 如需新增 table 通用区块或筛选交互，再改 `prototype.table.js`
8. 如需新增纯函数工具，再改 `prototype.utils.js`
9. 如需调整共享样式入口或 imports，再改 `prototype.css`
10. 如需调整共享壳样式、dashboard 样式或 table 样式，再改对应的共享 css 文件
11. 只有新增页面样式挂载机制时才改 `prototype.style-loader.js`
12. 只有新增启动或装配逻辑时才改 `prototype.js`

### 7.1 mock 持久化规则

- 默认存储 key 由 `namespace + project + version` 组成
- 首次打开页面时，框架会用对应页面模块中的示例数据自动 seed 一份本地 mock 数据
- 页面渲染时优先读取持久化数据；若本地无数据，则回落到页面模块与注册表合并后的默认 seed
- 当页面结构发生明显变化时，建议同步提升 `mockStore.version`，让本地 seed 重新初始化
- `localStorage` 适合刷新后仍保留的 mock 数据
- `sessionStorage` 适合只在当前会话内保留的 mock 数据

### 7.2 页面级交互规则

- 页面自己的局部交互优先写在对应的 `resources/scripts/pages/*.page.js`
- 页面模块如需挂载交互，可导出 `setup(context)`
- `setup(context)` 在页面渲染完成后执行，可返回清理函数用于下一次重渲染前释放事件绑定
- 通用导航和 toast 这类共享壳交互，仍放在 `prototype.shell.js`
- 表格筛选这类 table 共享交互，仍放在 `prototype.table.js`
- 只有当多个页面都要复用同一类能力时，才把逻辑上收进对应共享层，最终再由 `prototype.components.js` 装配

## 8. 新项目或新原型的改法

以后起一个新产品原型，优先只改这 3 类内容。

### 8.1 品牌和导航

在 `resources/scripts/prototype.config.js` 里改：

- `app`
- `navigation`

补充规则：

- 如果要做一级导航 + 二级导航，`navigation` 使用 `children` 数组
- `app.navSectionTitle` 可留空，留空后左侧不显示分组标题文案

### 8.2 页面内容

在下面三层里改：

- `resources/scripts/prototype.config.js` 中的 `pageRegistry`
- `resources/scripts/pages/*.page.js` 中的页面定义
- `resources/css/pages/*.css` 中的页面局部样式

补充规则：

- 如果只是当前页面自己的展示逻辑或交互，优先改该页面的 `*.page.js`
- 如果只是当前页面自己的样式，优先改该页面的 `resources/css/pages/*.css`

### 8.3 新增页面模板

当前仓库已内置两套新增页面模板：

- `templates/page/dashboard`
- `templates/page/table`

模板默认包含：

- 真实页面壳 html 模板
- `files/<页面>/data.js` 模板
- `files/<页面>/styles.css` 模板
- 页面模块 `*.page.js` 模板
- 页面样式 `*.css` 模板

新增页面时，优先从模板目录复制，不再默认从现有业务页面手工挑文件。

模板占位符说明见：

- `templates/page/README.md`

### 8.4 通用样式

优先改 `resources/css/prototype.*.css` 这些共享样式文件：

- `:root` 颜色变量
- 导航样式
- 卡片样式
- 表格样式
- 按钮样式

说明：

- 页面顶部默认只保留主标题
- 如果不需要小标题和说明，不必强依赖 `app.eyebrow` 或页面 `subtitle`
- `resources/css/prototype.css` 现在只是样式入口文件，主要负责 `@import`

## 9. 标准工作流

以后每次新增页面，建议固定按这个顺序：

1. 先判断新页面属于 `dashboard` 还是 `table`
2. 从 `templates/page/dashboard` 或 `templates/page/table` 复制模板文件
3. 统一替换模板中的 `__PAGE_NAME__`、`__PAGE_KEY__`、`__PAGE_TITLE__`、`__PAGE_PACKAGE_ID__`
4. 把模板文件分别落到真实路径：`<页面>.html`、`files/<页面>/`、`resources/scripts/pages/`、`resources/css/pages/`
5. 检查页面 html 中的共享脚本顺序是否保持模板默认顺序，不要重写 Axure 基座
6. 在 `prototype.config.js` 的 `pageRegistry` 中补页面注册信息
7. 在 `navigation` 中补一个同名 key 的菜单项
8. 在 `data/document.js` 中把页面挂到 sitemap
9. 如有必要，再改共享 `prototype.*.css`
10. 只有新增通用能力时，再改 `prototype.shell.js` / `prototype.dashboard.js` / `prototype.table.js` / `prototype.js`
11. 本地打开对应页面验证
12. 确认没问题后重新打包上传

## 10. 新增页面的最小检查清单

新增页面后，至少检查这 6 件事：

1. `navigation` 中是否有对应 key
2. `pageRegistry` 中是否有同名 key
3. 对应页面模块文件是否存在
4. 对应页面样式文件是否存在，且页面模块已声明 `styleSource`
5. `defaultPage` 是否仍然正确
6. 对应页面 html 是否能正常打开
7. 页面标题、按钮、表格或区块是否正常渲染
8. `data/document.js` 中是否已挂到正确目录
9. 模板占位符是否已全部替换完成

额外再检查 3 件事：

10. 页面文件名、`files/<页面>` 目录名、sitemap url 是否完全一致
11. 页面 html 和 `files/<页面>/data.js` 是否保持 UTF-8 BOM
12. 打包后的 zip 是否保留顶层目录

## 11. 打包上传

不要再用系统默认方式随意压缩。

默认打包方式如下：

```powershell
7z a -tzip mockplus-upload.zip .\mockplus-upload
```

其中 `mockplus-upload` 目录内应包含：

- `data`
- `files`
- `images`
- `plugins`
- `resources`
- `index.html`
- `start.html`
- `start_c_1.html`
- `start_with_pages.html`
- `仪表盘.html`
- `视频流分析.html`
- `视频流新建任务.html`

然后上传项目根目录下生成的文件：

- `mockplus-upload.zip`

说明：

- 使用 7-Zip 并保留顶层目录后，zip 结构更接近原始 Axure 导出包
- `mockplus-upload.zip` 为默认上传包
- 其他压缩方式和命名不作为默认流程

### 11.1 打包命名规则

后续统一使用：

- 目录名：`mockplus-upload`
- 压缩包名：`mockplus-upload.zip`

不使用 `7z`、`tar` 之类的区分性命名。
